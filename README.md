デンソーのハンディーターミナル向け検品アプリです。  
N88-BASIC を元にした独自 BASIC で書かれており、メイン処理は `_Org/main.src` の `MAIN:` から始まります。

## 概要

このアプリは、現品票 QR、部品展開表 QR、社員 QR、日付 QR などをスキャンし、検品・ピッキング・出荷検査・日報入力・在庫操作・棚卸・山寺工場向け処理・シスメックス向け処理を行います。

処理結果は主に `TKSDATA.CSV` に蓄積し、メニューの「データ送信」から FTP でサーバへ送信します。一部の在庫系処理はソケット通信でサーバへ即時連携します。

## 起動処理

`MAIN:` では次の初期化を行います。

1. 画面を小フォントに設定する。
2. プリンタ、ラベル、ログ、税率、印刷ステータスなどの設定値を `Sts.SetVal` で初期化する。
3. Bluetooth プリンタ設定を初期化する。
4. 起動に必要な `pTenPr.dat` と `pYmdPr.DAT` の存在を確認する。
5. 機種が `BHT13QWB` または `BHT80QW` の場合は無線を使用する設定にする。
6. 無線利用時は Wi-Fi、FTP、ソケット通信先を設定し、最初に `Soc.Live` で通信チェックを行う。
7. `gMenu1$` でメニューを表示し、選択された処理へ分岐する。

## 実績データ

`TKSDATA.CSV` は固定長レコード扱いで、`fData.src` の `Dat.Put` により 1 レコードずつ追記されます。

主なレコード種別:

| 種別 | 作成元 | 内容 |
|---|---|---|
| `F3CHK_2` | `gTenSc$` | 展開表3点照合の OK/NG/警告 |
| `SYUKA` | `gSyuka$` | 出荷検査の OK/NG/警告 |
| `PICKING` | `gPikng$`, `gPikSuu$` | ピッキング実績 |
| `Nippo1` | `gNippo$` | 日報入力 |
| `NipTm1` | `gNipTm$` | 作業時間入力の日報 |
| `TanZik` | `ztn.Copy` | 在庫棚卸データ |

`gSnd0$` は `TKSDATA.CSV` を `YYMMDD-BHTID.CSV` というサーバ側ファイル名で FTP アップロードします。送信に成功すると `Dat.Clear` でローカルの `TKSDATA.CSV` をクリアします。

## 通信

### FTP

FTP は主にファイル送受信に使われます。

| 処理 | 内容 |
|---|---|
| データ送信 | `TKSDATA.CSV` をアップロードする |
| マスタ受信 | `kosei.txt` をダウンロードする |
| 時刻補正 | `timefile.txt` をダウンロードし、端末時刻を設定する |
| 保守ダウンロード | `SETUPBCD.PD4`, `LDFTP.PD4`, `TKSTANA.PD4`, `TRANS.PD4` を受信する |

### ソケット通信

在庫系、山寺、シスメックスの処理では `Sock.Param` に `^` 区切りでパラメータを組み立て、`Sock.Send` / `Sock.Recv` でサーバと通信します。

主な呼び出し名:

| 呼び出し名 | 用途 |
|---|---|
| `tksLive` | 起動時の通信チェック |
| `tksZkNyu` | 棚入れ/ピッキング入庫 |
| `tksZkInq` | 在庫数問合 |
| `tksZktn` | 棚卸数送信 |
| `tksZkChg` | 在庫調整 |
| `tksRiyu` | 在庫調整理由取得 |
| `YmdNyu` | 山寺入庫 |
| `SmxNyu` | シスメックス入庫 |
| `SmxSyu` | シスメックス出荷 |
| `SmxNyuItem` | 通常QRの品目マスタ、または`$Ny`の元連番から親品目・品名・数量・洗浄方法を照会 |
| `SmxNyuSheet` | 通常QRの入力洗浄方法0をシート登録。`$Ny`の洗浄方法0では終了更新とシート登録を同一トランザクション実行 |
| `SmxNyuFinish` | `$Ny`の洗浄方法0以外で`T_SmxTrc.終了日時`だけを更新 |

### ソケット電文の共通仕様

送信文字列は `Sock.Param.Init(呼び出し名)` で先頭に呼び出し名を設定し、その後 `Sock.Param.Add` で項目を追加します。項目は `^` 区切りです。

例:

```text
呼び出し名^BHTID^BHT日付^BHT時間^社員CD^納入日/日付^判定または業務項目...
```

送信後は `Sock.Recv` で応答を受け取り、基本的に次の項目を参照します。

| 応答項目 | 内容 |
|---|---|
| `Sock.Recv.Items$(0)` | 処理結果コード。`0` が正常 |
| `Sock.Recv.Items$(1)` | サーバ側メッセージ |
| `Sock.Recv.Items$(2)` | 在庫系では在庫数。`Soc.Live` では日付 |
| `Sock.Recv.Items$(3)` | `Soc.Live` では時刻 |

`Soc.Send` と `Soc.YmdSend` は送信前に電文を `sp-send.txt` へ保存します。送信成功後に先頭 1 件を削除し、まだ残っている場合は続けて送信します。通信失敗時は最大 3 回まで再接続してリトライします。

### `Soc.Live` の送信内容

起動時の通信チェックです。正常応答時、サーバから返る日付・時刻が端末日付と大きくずれていなければ端末の `date$` / `time$` を補正します。

| 順番 | 項目 | 値 |
|---:|---|---|
| 1 | 呼び出し名 | `tksLive` |
| 2 | BHTID | `SysBht$` |
| 3 | BHT日付 | `Date$` |
| 4 | BHT時間 | `Time$` |
| 5 | バージョン | `ver1$ + "." + ver2$` |
| 6 | 納入日 | 空 |
| 7 | 判定 | 空 |

### 在庫系送信内容

`Soc.Zaiko`, `Soc.ZaikoTm`, `Soc.ZkChg` は最終的に `Soc.Send` を呼びます。電文の基本形は次の通りです。

| 順番 | 項目 | 値 |
|---:|---|---|
| 1 | 呼び出し名 | `tksZkNyu` / `tksZkInq` / `tksZktn` / `tksZkChg` |
| 2 | BHTID | `SysBht$` |
| 3 | BHT日付 | `dt$` |
| 4 | BHT時間 | `tm$` |
| 5 | 社員CD | `sya.cd$` |
| 6 | 納入日 | `sya.dt$` |
| 7 | 判定 | 空 |
| 8 | 品番 | `hno$` |
| 9 | 図番 | `zno$` |
| 10 | 入数 | `nsu$` |
| 11 | 出数 | `ssu$` |
| 12 | 理由区分 | `kbn$` |

呼び出し元ごとの実引数:

| 画面/処理 | 呼び出し | 品番 | 図番 | 入数 | 出数 | 理由区分 | 備考 |
|---|---|---|---|---|---|---|---|
| ピッキング | `tksZkNyu` | 空 | `pck.SakiCd$` | `0` | 現品票総数 `gt2.qrsum$` | 空 | 照合先を初回スキャンした時だけ送信 |

| 在庫数問合 | `tksZkInq` | 空 | `GQR.zro$` | `0` | `0` | 空 | 応答の `Items$(2)` を在庫数として表示 |
| 棚入れ(入庫) | `tksZkNyu` | 空 | `GQR.zro$` | `GQR.sum$` | `0` | 空 | 現品票の総数を入庫数として送信 |
| 棚卸の事前照会 | `tksZkInq` | 空 | `GQR.zro$` | `0` | `0` | 空 | 現在数表示用 |
| 棚卸の登録 | `tksZktn` | 空 | `GQR.zro$` | 手入力数量 `suu$` | 空 | 空 | 棚卸数を送信 |

### SMX戻り品入庫（`$Ny`）

`$Ny`は6項目固定です。数量はQR値を使用し、洗浄方法はQR値を初期表示した後、BHTの入力値を登録判定に使用します。通常QRは品目マスタの洗浄方法を初期表示します。

| CSV位置 | 内容 | BHT内の保持先 |
|---:|---|---|
| 1 | `$Ny` | QR種別判定 |
| 2 | 出庫先 | `Smx.Dst$` |
| 3 | 元の連番 | `Smx.Seq$` |
| 4 | 子品目 | `Smx.Cd2$` |
| 5 | 数量 | `Smx.Suu$` |
| 6 | 洗浄方法 | `Smx.Wash$` |

3通信の送信項目は共通です。

```text
呼出名^BHTID^BHT日付^BHT時間^社員CD^納入日^QR全文^出庫先^元連番^子品目^数量^洗浄方法^備考
```

処理順は次のとおりです。

1. `SmxNyuItem`で通常QRは品目マスタ、`$Ny`は元連番を検索し、商品情報を確認する。
2. 応答の親品目、品名、数量、マスタ洗浄方法を画面へ反映する。`$Ny`の数量差異は警告するが登録可能とする。
3. 通常QRだけ数量・洗浄方法を入力する。入力洗浄方法が数値0なら`SmxNyuSheet`で`T_シートリーダ`へ追加する。
4. 通常QRで入力洗浄方法が0以外なら、既存`SmxNyu`で`T_SmxTrc`へ登録する。正常時は「洗浄へ出庫です。」「管理番号 n で登録しました」を表示する。
5. `$Ny`はQRの数量・洗浄方法を編集しない。QR洗浄方法が0なら`SmxNyuSheet`で終了日時更新とシート登録を同一トランザクション実行し、0以外なら`SmxNyuFinish`で終了日時だけを更新する。
6. `$Ny`の正常時は「入庫を登録」と表示する。登録処理側でも元レコードまたは品目マスタを再検索し、登録条件を再検証する。

品名照会の応答は次の順です。最後にはSocketサーバーがストアドのRETURN値を追加します。

```text
結果コード^メッセージ^親品目^品名^元数量^マスタ洗浄方法^ストアドRETURN値
```

## QR/バーコードの扱い

`qrGenpin.src` の `GetQr` が現品票 QR を解析します。

| 先頭 | 判定 | 主な取得項目 |
|---|---|---|
| `Z` | 現品票 | 品番、図番、累進、図番累進、総数、品名 |
| `T01` | 新EDI系の現品票 | 図番、累進、図番累進、総数 |
| `T` | 部品展開表 | 図番、累進、図番累進 |

社員/納入日 QR は `gSyain.src` で読み取ります。

| 先頭 | 用途 |
|---|---|
| `A` | 社員コード |
| `B` | 納入日 |

日報では、社員 `A`、場所 `B`、工程 `K`、時間 `H` の QR/コードを扱います。

## 構成マスタと必須チェック

`kosei.txt` は構成マスタです。項目幅は `fKosei.src` で次のように定義されています。

| 項目 | 幅 | 内容 |
|---|---:|---|
| 親図番 | 16 | 親の図番累進 |
| 子図番 | 16 | 子の図番累進 |
| 員数 | 5 | 必要数 |
| 品名 | 30 | 品名 |
| 区分 | 4 | 必須チェック対象かどうか |
| 予備 | 6 | 予備 |

出荷検査では `Ksi.set` が対象親図番の必須レコードを `koseiWk.txt` にコピーし、スキャン後に `Ksi.del` で消し込みます。

ピッキングでは `pck.init` が対象親図番の構成を `koseiPcw.txt` にコピーします。照合元スキャンは `pck.scanMoto`、照合先スキャンは `pck.scanSaki` で回数管理され、必須品が未検品の場合は `HissuList2$` で警告します。

## プリンタ

Bluetooth プリンタは `funcAll.src` の `St.*` / `Bt*` 系処理で制御します。  
ラベル定義ファイルを読み込み、`{@1}` から `{@9}` のようなプレースホルダを `St.Items$()` の値で置換して印刷データを送信します。

起動時の初期プリンタ設定は `PW208` を前提に設定されています。プリンタステータスは `PT208` 系と `PW208` 系それぞれで `Sts.SetVal` に登録され、エラー時の表示やリトライ/終了判定に使われます。

## 保守時の注意

- ソースは Shift-JIS/CP932 です。編集時に UTF-8 へ変換すると、BASIC 開発環境や実機で文字化けする可能性があります。
- `main.src` の `$INCLUDE` 順に依存しているため、共通変数や関数を移動する場合は include 順も確認してください。
- `TKSDATA.CSV` は送信成功後にクリアされます。送信前の実績確認や障害調査では、クリア前の内容を確認してください。
- `Dat.Count > 500` または処理により `Dat.Count > 5000` の場合、作業を止めてデータ送信を促します。
- 無線機種以外では、ソケット/FTP 前提の一部メニューは使用できず、メニューへ戻る動きになります。
- Wi-Fi、FTP、ソケット接続先の設定値は `main.src` にハードコードされています。環境変更時は実機設定だけでなくソース側の初期値も確認してください。
- 在庫棚卸は作業途中データを `tnzaiko.txt` に持ちます。重複伝票番号のチェックやメニュー復帰時の `TKSDATA.CSV` コピー動作に注意してください。
- プリンタ関連は `LifePd.DAT` の設定値と Bluetooth アドレスに依存します。プリンタ未設定時はメニュー下部に `PR**` / `AD**` が表示されます。
