# SMX入庫 Web仕様対応

## 文書区分

| 項目 | 内容 |
|---|---|
| MD分類 | 改修 |
| 実施日 | 2026-06-21 |
| 対象 | BHTのSMX入庫、Socket通信、SQL Serverストアド |

## 調査内容

| 調査対象 | 確認結果 |
|---|---|
| BHTの従来SMX入庫 | カンマ区切りQRから親・子・数量を取得し、`SmxNyu`を1回送信していた |
| 既存共通電文 | `Soc.YmdSend`が呼出名、BHTID、BHT日付、BHT時間、社員CD、納入日を先頭に付加する |
| Socket応答 | `^`分割後、位置0が結果、位置1がメッセージ、位置2以降が追加OUTPUT |
| 通常QR | 品目マスタの洗浄方法を初期表示し、BHTで確定した入力値が0ならシートリーダへ登録する |
| `$Ny`仕様 | 6項目固定。数量・洗浄方法は表示のみとし、現行は洗浄方法に関係なく`SmxNyu`で終了日時更新とシートリーダ登録を行う |

| `_Org/gSmxNyu.src` | 通常QRだけに数量・洗浄方法入力画面を追加し、`$Ny`はQRの数量・洗浄方法を表示のみとするよう修正 |

## 通信仕様

| 呼出名 | 用途 | 正常時の主なOUTPUT |
|---|---|---|
| `SmxNyuItem` | 品名問い合わせ・登録前確認 | 結果、メッセージ、親品目、品名、元数量、マスタ洗浄方法 |
| `SmxNyu` | 登録本処理。通常QRは洗浄方法で登録先分岐、`$Ny`は消込＋シート登録 | 結果、メッセージ |

## 判断理由

| 判断 | 理由 |
|---|---|
| 登録本処理を`SmxNyu`へ集約 | 提示電文の呼出名に合わせ、登録先分岐をストアド側にまとめるため |
| 登録前にストアド側で再検証 | 品名照会後から登録までの間に他端末が処理する競合を防ぐため |
| 通常QRはBHT入力値で判断 | QR値やマスタ値は初期表示に限定し、作業者が確定した値を登録条件とするため |
| `$Ny`はQR値を変更しない | Web版と同じく、返却QRに記録された数量・洗浄方法を表示用・登録分岐用として扱うため |
| 通常QRの入力値0はシート登録経路 | 洗浄方法0でも`T_シートリーダ`へ実績を残すため。現行登録送信は`SmxNyu`に集約 |
| `$Ny`は同一トランザクション | 終了更新またはシート登録の一方だけが残る状態を防ぐため |
| 不要ストアドを削除 | `SmxNyuSheet`、`SmxNyuFinish`の役割を`SmxNyu`へ統合したため |

## 不具合対応

| 項目 | 内容 |
|---|---|
| 現象 | `Soc.YmdSend`で例外が発生すると、別関数の終了ラベルへ復帰する可能性があった |
| 原因 | エラーハンドラが`resume socSend.End`を指定していた |
| 影響範囲 | 山寺入庫、従来SMX入庫・出荷、新しいSMX入庫3通信 |
| 対応 | `resume socYmdSend.End`へ修正 |
| 回避方法 | 修正版を配布するまで、通信が安定した環境で使用し、異常時はDB登録結果を確認する |

### 洗浄方法0でシートリーダ実績が作成されない不具合

| 項目 | 内容 |
|---|---|
| 現象 | 通常QRで洗浄方法に0を入力しても`T_シートリーダ`にデータが作成されない |
| 原因 | 通常QRが新しい`SmxNyuSheet`を通らず、処理分岐もBHTの入力値を受け付けていなかった |
| 影響範囲 | SMX入庫で通常QRを読み取り、洗浄方法0として登録する端末・利用者 |
| 対応 | 洗浄方法入力画面を追加し、入力値0のとき`SmxNyuSheet`を呼ぶよう修正 |
| 回避方法 | 修正版配布までは対象QRの登録結果をDBで確認し、必要に応じて管理者が実績を補完する |

### `$Ny`の終了更新とシート登録が片方だけ残る不具合

| 項目 | 内容 |
|---|---|
| 現象 | `$Ny`の洗浄方法0で、通信や後続登録の失敗により終了日時更新だけが残る可能性がある |
| 原因 | BHTから終了更新とシート登録を別々に実行すると、2処理を一括でロールバックできない |
| 影響範囲 | `$Ny`戻り品QRのうち、洗浄方法0を使用するSMX入庫端末・利用者 |
| 対応 | `SmxNyuSheet`内で終了日時更新とシート登録を同一トランザクション実行し、どちらかが失敗した場合は両方をロールバックする |
| 回避方法 | 修正版配布までは`T_SmxTrc.終了日時`と`T_シートリーダ`の両方を確認する |

## 2026-06-22 通常QRの品名照会仕様変更

| 項目 | 内容 |
|---|---|
| 種別 | 改修 |
| 調査内容 | 通常QRの`SmxNyuItem`は`T_data54`に該当がなくても正常応答し、空の洗浄方法から手入力できる実装だった |
| 改修内容 | `SmxNyuItem.sql`で`T_data54`の該当行有無を判定し、通常QRでデータがなければ結果`1`・「品名なし」を返すよう変更 |
| 改修内容 | データがある場合は従来どおり品名と`T_data54.洗浄方法`を返し、BHTの洗浄方法入力初期値に使用 |
| 影響範囲 | SMX入庫の通常QR読取後の品名照会と洗浄方法入力。`$Ny`の元連番照会・登録分岐は変更しない |
| 判断理由 | 通常QRはDBに登録された対象品だけを入庫し、マスタの洗浄方法を入力初期値とする業務仕様のため |
| 未対応 | 変更後SQLのSQL Serverへの適用、Socket経由のBHT実機テスト |
| 保留 | 本番`T_data54`に品名が空文字の行が存在する場合の扱い |

## 2026-06-27 ストアド応答msg短縮・エラーresult方針

| 項目 | 内容 |
|---|---|
| 種別 | 改修 |
| 調査内容 | `SmxNyuItem`、`SmxNyuSheet`、`SmxNyuFinish`の`@msg`を確認し、BHT画面表示には長い文言が含まれていた |
| 改修内容 | エラー・正常時の`@msg`を「品名なし」「元連番不正」「シート登録失敗」「入庫登録」など短文へ変更 |
| 改修内容 | `02_設定/ストアド/README.md`へ、`@result=0`を正常、エラー時は0以外の正の整数、ストアドRETURN値もエラー時は`@result`、`@msg`は漢字10文字・半角20文字程度までとする方針を追記 |
| 判断理由 | ハンディターミナルの小さい画面へ表示するため、ストアド応答msgは短い文言に統一する必要がある |
| 影響範囲 | SMX入庫の品名照会、シートリーダ登録、終了日時更新のSocket応答とBHT画面表示 |
| 未対応 | SQL Serverへの変更後ストアド適用、BHT実機での表示確認 |
| 保留 | SocketサーバーはストアドRETURN値を末尾付加するため、BHT側の既存判定箇所が`@result`とRETURN値のどちらを使用してもエラー時は非0になる |

## 2026-06-27 出庫先エラーmsg・result整理

| 項目 | 内容 |
|---|---|
| 種別 | 改修 |
| 調査内容 | `SmxNyuItem`、`SmxNyuSheet`、`SmxNyuFinish`で、出庫先不一致と商品不一致が同じ`result=2`・「商品が違います」に混在していた |
| 改修内容 | `destination`が`1000020880`ではない場合、または`$Ny`元レコードの出庫先と一致しない場合は、`result=6`・「出庫先が違います」を返すよう変更 |
| 改修内容 | 子品目・親子品目の不一致または不足は従来どおり`result=2`・「商品が違います」とし、意味の同じエラー番号を統一 |
| 改修内容 | `02_設定/ストアド/README.md`へresultコード表を追加 |
| 判断理由 | BHT画面で作業者が出庫先違いと商品違いを判別できるようにし、Socket応答の番号も原因別に揃えるため |
| 影響範囲 | SMX入庫の品名照会、シートリーダ登録、終了日時更新のSocket応答とBHT画面表示 |
| 未対応 | SQL Serverへの変更後ストアド適用、BHT実機での表示確認 |
| 保留 | `result=1`は照会対象なし・照会キー不正として既存の複数メッセージを維持 |

## 2026-06-27 `SmxNyu`登録本処理の一本化

| 項目 | 内容 |
|---|---|
| 種別 | 改修 |
| 調査内容 | BHTの通常QR登録は`SmxNyu^BHTID^BHT日付^BHT時刻^社員CD^納入日^QR全文^子@親^^数量^0^0^洗浄方法`形式で送信される |
| 改修内容 | `02_設定/ストアド/SmxNyu.sql`を、通常QR・洗浄方法000以外は`T_SmxTrc`追加、通常QR・洗浄方法000は`T_シートリーダ`追加、`$Ny`は元`T_SmxTrc.終了日時`更新＋`T_シートリーダ`追加に変更 |
| 改修内容 | `_Org/gSmxNyu.src`の登録送信を`SmxNyuSheet`／`SmxNyuFinish`分岐から`SmxNyu`一本に変更 |
| 改修内容 | `02_設定/ストアド/README.md`と`02_設定/設定_SMX入庫SocketサーバーINI.md`へ、`SmxNyu`電文・分岐仕様を追記 |
| 判断理由 | Socketサーバー側の呼出名を`SmxNyu`に集約し、洗浄方法とQR種別によるDB登録先の判断をストアド側へ寄せるため |
| 影響範囲 | SMX入庫登録時のSocket電文、通常QRの`T_SmxTrc`／`T_シートリーダ`登録、`$Ny`戻り品QRの消込とシート登録 |

| 保留 | `T_SmxTrc`の完全DDLが本リポジトリにないため、`SmxNyu.sql`は現行SQLで参照済みの列と`洗浄方法`列を前提にしている |

## 2026-06-27 `T_SmxTrc`実スキーマ反映

| 項目 | 内容 |
|---|---|
| 種別 | 改修 |
| 調査内容 | 実DBの`T_SmxTrc`は`ID`がIDENTITY主キー、`tm`が既定値、`自IPアドレス`・`現在日付`・`BHT側時刻`がNOT NULLだった |
| 改修内容 | `SmxNyu.sql`の通常QR・洗浄方法000以外の`T_SmxTrc` INSERT列を実スキーマに合わせて拡張 |
| 改修内容 | `自IPアドレス`には利用者指定どおり社員NO、`現在日付`にはBHT日付、`BHT側時刻`にはBHT時刻を登録 |
| 改修内容 | QR 4項目目の棚番、6項目目の所要日、9項目目以降のプラント・伝票番号・ロット・保管場所、QR全文、開始日時、備考を登録対象へ追加 |
| 判断理由 | `ID`はDB内部主キーでINSERT対象外、`連番`はBHTへ返す管理番号として採番・登録するため |
| 影響範囲 | 通常QRで洗浄方法が000以外のSMX入庫登録。洗浄方法000のシート登録と`$Ny`消込経路は登録先が異なるため直接影響しない |
| 未対応 | SQL Serverへの変更後`SmxNyu.sql`適用、BHT実機・DB結合確認 |
| 保留 | `自IPアドレス`列名は実IPを示すが、現行仕様では社員NOを格納する |
