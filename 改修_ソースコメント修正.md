## 調査内容

実装とコメントを照合し、コメントの誤記、古い電文説明、処理名の不一致、同じ粒度の説明における表記ゆれを確認した。

| 調査観点 | 確認内容 |
|---|---|
| 処理目的 | 関数名・呼出元・実際の処理と見出しコメントが一致するか |
| 引数・戻り値 | コメントの項目名、順番、正常値が実装と一致するか |
| Socket電文 | `Sock.Param.Init` / `Sock.Param.Add` の追加順と番号が一致するか |
| 状態・タイミング | 初回処理、定期処理、削除、初期化等の説明が実際のタイミングと一致するか |
| コメント表記 | BASICのアポストロフィコメントにC系の `//` が混在していないか |

### 粒度・表記統一

| 対象 | 統一内容 |
|---|---|
| 全36ファイル | コメント開始をBASICのアポストロフィ形式へ統一し、`'//` の混在を解消 |
| 処理見出し | 「何をする処理か」を1文で記載 |
| 公開的な関数説明 | 必要な箇所は引数、戻り値、送信項目まで記載 |
| 行末コメント | 電文位置や値の意味など、コードだけでは判断しにくい内容を記載 |
| 区切り線 | 処理単位の区切りを同じアポストロフィ形式へ統一 |

## 判断理由

| 判断 | 理由 |
|---|---|
| コメントだけを変更 | ユーザー指定の範囲を厳守し、実行動作への影響を避けるため |
| 既存コメントを全削除・全面再作成しない | 過去の変更理由や機種固有情報を保持するため |
| 電文コメントは全体位置で採番 | 共通項目を含む実際の送信順を追跡しやすくするため |
| 自明な代入の説明を増やさない | コメント粒度を処理目的・判断条件・外部仕様中心に揃えるため |
