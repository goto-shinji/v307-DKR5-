# 静的調査で確認した問題一覧

## 文書区分

| 項目 | 内容 |
|---|---|
| MD分類 | 不具合 |
| 記録日 | 2026-06-21 |
| 対象 | ツカサ検品照合 BHTアプリ v3.07 |
| 確認方法 | ソースコードの静的調査 |
| 実機確認 | 未実施 |

本書は既存の各「調査」MDで確認した問題を、不具合記録の必須項目である「現象・原因・影響範囲・回避方法」に揃えた一覧です。原因の詳細と根拠は各調査MDを参照してください。

画面・機能単位で参照する場合は、`06_不具合/不具合_画面・機能別既知不具合一覧.md`を参照してください。

## Socket通信

| 現象 | 原因 | 影響範囲 | 回避方法 |
|---|---|---|---|
| 無線無効時に未送信でも正常扱いになる | `SysWifi=false` の場合に通信せず `0` を返す | 在庫調整、在庫問合、棚入れ、棚卸、山寺、SMX、LIVE | 無線有効端末だけで実行し、処理後にサーバ登録結果を確認する |
| 応答欠落後の再送で二重登録になり得る | 送信済みでも応答が失われると `sp-send.txt` に電文が残り、取引IDもない | Socket登録を行う全業務 | 通信異常後は再操作前にサーバ登録と `sp-send.txt` の残件を確認する |
| 接続失敗でも正常値を返し得る | `Sock.Conn` が例外を戻り値へ設定しない | Socket通信を使う全画面 | 業務開始前にLIVE疎通を確認し、登録後はサーバ側結果を照合する |
| 受信失敗時に前回応答を参照し得る | 実データ受信時以外は `Sock.Recv.Items$()` をクリアしない | 在庫数表示、結果メッセージ、LIVE日時 | 通信異常後はアプリを再起動し、前回応答が残らない状態で再確認する |
| 受信失敗を再試行しない | 再試行対象が送信エラーに限られる | Socket通信を使う全画面 | 受信異常時は登録結果を確認してから再操作する |
| 応答項目不足で固定添字を参照する | `Sock.Recv.ItemCount` を確認しない | 在庫数、メッセージ、LIVE日時の表示 | サーバ応答形式を変更せず、異常表示時は処理を中止する |
| ピッキングの指定日時が送信されない | `Soc.ZaikoTm` が引数でなく送信時の端末日時を使う | ピッキング、数量ピッキング | 端末日時を正しく合わせ、通信遅延後の登録日時を確認する |
| 長い電文が切れる可能性がある | `Sock.Param$` が255文字固定で事前長チェックがない | 元スキャンを含む山寺・SMX電文 | 長いQRの使用を避け、サーバ側の受信値を確認する |

## FTP通信

| 現象 | 原因 | 影響範囲 | 回避方法 |
|---|---|---|---|
| 無線同期失敗の原因が後段エラーとして表示される | `Sock.Sync` の戻り値を判定せずFTP接続へ進む | データ送信、マスタ受信、時刻補正、保守ダウンロード | Wi-Fi接続状態を事前確認し、同期失敗時は端末を再接続してから実行する |
| 再試行成功後もエラー扱いになり得る | 最初の例外値を成功時に `0` へ戻さない | FTPを使用する全処理 | 画面結果だけでなく、送受信先ファイルの有無・更新日時を確認する |
| FTP API失敗を見落とし得る | APIの `reply%` を評価しない | FTPを使用する全処理 | サーバ上のファイルと端末側ファイルを処理後に照合する |
| CSV再送で実績が重複し得る | 追加モード送信で一意な送信IDがない | `TKSDATA.CSV` の実績取込 | 異常後はサーバ側ファイルを確認し、受信済みなら安易に再送しない |
| 時刻補正失敗が実績送信失敗に見える | CSV消去後に `timefile.txt` を受信する一連処理になっている | データ送信画面、端末時刻 | サーバ側CSVを確認し、実績送信済みならCSV復元・再送をしない |

## Bluetoothプリンタ通信

| 現象 | 原因 | 影響範囲 | 回避方法 |
|---|---|---|---|
| 接続失敗でもプリンタ選択成功に見える | `BtCheck` が `St.Open` 失敗を戻り値へ設定しない | プリンタ選択、以後の全印刷 | 選択直後にテスト印刷し、実際の出力を確認する |
| 未印刷でも成功扱いになり得る | 定義ファイル異常、ACKタイムアウト、NAK終了時に非0を設定しない | 展開票、山寺現品票、フィード等 | 印刷物を目視確認し、出ていなければプリンタ状態と定義ファイルを確認する |
| PT208詳細設定が送信されない | 空の `filename$(1)` が実際の読込対象になる | PT208の詳細設定 | 詳細設定後にプリンタ本体の設定値を確認する |
| 機種判定失敗後も初期設定を続ける | `St.setAdr` エラー後に即終了せず、後続結果がエラーを上書きし得る | プリンタ選択、機種別初期化 | 対象プリンタだけを選び、選択後に機種表示と印刷結果を確認する |
| 別機器をプリンタと誤判定し得る | 応答内容ではなく受信長だけで機種判定する | Bluetooth検索、プリンタ選択 | BDアドレスを照合し、周辺検索では対象機器を特定して選ぶ |
| スキャンしたBDアドレスの形式を検証しない | 手入力時と異なり12桁・16進文字の確認がない | プリンタ選択 | 12桁の正しいBDアドレスを手入力するか、読取後の表示を照合する |
| 接続中の別プリンタへ送る可能性がある | 既オープン状態を接続先確認なしで正常扱いする | 全ラベル印刷 | プリンタ交換時はアプリまたは端末を再起動してから選択し直す |
| 印刷失敗を画面で通知できない | 印刷画面が印刷関数の戻り値を使用しない | 展開票、山寺現品票、フィード等 | 必ず印刷物を目視確認する |

## バーコード・入力・画面処理

| 現象 | 原因 | 影響範囲 | 回避方法 |
|---|---|---|---|
| 在庫問合で社員CD・納入日が空で送信される | 共通入力後に値を空文字へ戻し、キャンセル結果も確認しない | 在庫数問合 | サーバ側で空項目を許容する運用に限定し、操作後に照会結果を確認する |
| SMX出荷で社員入力をキャンセルしても次画面へ進む | `gSyacd$` の戻り値を確認せず状態を上書きする | SMX出荷 | キャンセル後は次画面で登録せず、メニューへ戻るまで操作を中止する。SMX入庫は2026-06-21に修正済み |
| 不完全な `Z` 現品票を受理し得る | 最低長84文字に対し総数・品名は85～105文字目を参照する | 出荷、ピッキング、在庫系、印刷、在庫棚卸 | 規定長105文字以上の正規QRだけを使用する |
| 短いQRから空項目を抽出し得る | 各画面の長さ確認が実際の抽出位置より短い | 3点照合、出荷、単純照合、ピッキング、山寺印刷 | 正規発行されたQRだけを使用し、画面表示項目を確認する |
| `T01` から画面ごとに異なる図番を抽出し得る | `T01` 専用解析と一般 `T` 判定が混在する | 山寺、3点照合、出荷、数量ピッキング等 | 対象画面の想定帳票だけを使用し、抽出図番を目視確認する |
| 不正な日付・時刻を受理し得る | 先頭文字と長さ中心で数値範囲を検証しない | 社員・納入日、日報、SMX出荷 | 正規QRを使用し、端末表示の日付・時刻を確認する |
| 数量0の扱いが画面ごとに異なる | 業務別の入力チェックが統一されていない | 在庫調整、棚卸、棚入れ、日報、山寺、SMX等 | 業務上0が有効か確認できない場合は0を登録しない |
| 従来SMX入庫票以外を誤認し得る | 従来QR互換分岐はカンマ項目数中心で固定識別子を確認しない | SMX入庫の従来QR分岐 | SMX入庫用に発行されたQRのみ使用し、親子品番表示を確認する。`$Ny`は厳密チェック済み |

## 対応済み

| 対応日 | 現象 | 原因 | 影響範囲 | 対応 |
|---|---|---|---|---|
| 2026-06-21 | 山寺・SMX送信例外時に誤った終了先へ復帰する | `socYmdSend.Err`が`resume socSend.End`を指定 | 山寺入庫、SMX入庫、SMX出荷 | `resume socYmdSend.End`へ修正 |
| 2026-06-21 | SMX入庫で社員入力キャンセル後も次画面へ進む | `gSyacd$`の戻り値を上書き | SMX入庫 | 戻り値が空でない場合はSMX入庫を終了するよう修正 |

## 判断理由

| 判断 | 理由 |
|---|---|
| ソース改修は行わない | 今回の依頼はMD再確認とカスタム指示への適合であり、業務仕様および実機試験結果が未確定のため |
| 不具合を本書へ集約する | 調査MD内の問題表には影響の記載はあったが、必須項目の回避方法が不足していたため |
| 回避方法は暫定運用とする | 静的調査のみであり、修正効果や実機挙動を検証していないため |
