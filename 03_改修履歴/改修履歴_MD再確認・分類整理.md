## 判断理由

| 判断 | 理由 |
|---|---|
| 既存の調査MDを削除・分割しない | 通信手順と問題発生経路を一続きで追える調査資料として有用なため |
| 不具合一覧を別ファイルにする | カスタム指示が求めるMD分類と、不具合記録の必須4項目を明確に満たすため |
| READMEを文書索引にする | 各MDの分類と用途を最初に判断でき、記録先の混同を防げるため |
| MD更新ルールをREADMEへ追記する | 作業前後に参照されやすい入口文書へ記録条件と分類を置くことで、記録漏れを防ぐため |
| 今回の作業記録を改修履歴へ追記する | ソースや設定ではなくMD運用ルールの整理であり、作業記録として追跡する内容のため |
| `改修履歴`フォルダーを分離する | カスタム指示では「改修」と「改修履歴」が別区分のため、改修内容と作業記録を混在させないため |
| `ストアド`を`02_設定/ストアド`へ移動する | ストアド配置・Socketサーバー設定仕様は設定区分の管理対象であり、分類外フォルダーを残さないため |
| テスト仕様書を仕様分類へ変更する | 内容がテスト項目、期待結果、DB・通信確認欄であり、調査結果より仕様書として扱う方がカスタム指示の分類に合うため |
| 空の調査MDを残さない | 同名の実体文書が別フォルダーにあり、空ファイルを残すと参照先や更新先を誤るため |
